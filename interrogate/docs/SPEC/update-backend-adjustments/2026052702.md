# `/ws/asr` 服务器连接失败检查结果（2026-05-27）

## 1. 现象复述

当前给出的现象是：

```text
本机：ws://127.0.0.1:5000/ws/asr 可以连接
服务器：wss://sx.juanquan.top/ws/asr 连接失败
浏览器控制台：WebSocket connection to 'wss://sx.juanquan.top/ws/asr' failed
```

服务器 Nginx 当前只为 `/ws/face` 单独配置了 WebSocket Upgrade：

```nginx
location /ws/face {
    proxy_pass http://sx;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "Upgrade";
    ...
}
```

没有看到 `/ws/asr` 或通用 `/ws/` 的 WebSocket 代理配置。

## 2. 当前代码链路

从当前仓库代码看，`html-interrogate` 的 ASR 主链路不是浏览器直连外部公网 ASR 服务，而是：

```text
浏览器页面
  -> buildWebSocketUrl('/ws/asr')
  -> 当前站点 WebSocket：wss://sx.juanquan.top/ws/asr
  -> Flask-Sock 后端 app/routes/ws/register.py:/ws/asr
  -> 后端 ASRClient
  -> DashScope 公网 WebSocket：wss://dashscope.aliyuncs.com/api-ws/v1/inference
```

证据：

- `html-interrogate/src/composables/useAsrSocket.ts` 使用 `new WebSocket(buildWebSocketUrl('/ws/asr'))`。
- `html-interrogate/src/utils/websocket.ts` 在 HTTPS 页面下会推导为 `wss://当前 host`。
- `app/routes/ws/register.py` 同时注册了 `/ws/face` 和 `/ws/asr`。
- `app/routes/ws/audio.py` 收到前端音频包后调用 `_asr_client.send(...)`。
- `app/service/audio/qwen_asr.py` 默认上游是 `wss://dashscope.aliyuncs.com/api-ws/v1/inference`，也可通过 `ASR_DASHSCOPE_ENDPOINT` 覆盖。

因此，浏览器控制台里的 `wss://sx.juanquan.top/ws/asr failed` 首先说明浏览器到项目后端的 WebSocket 握手没有成功；这一步还没有进入 DashScope 公网 WebSocket 代理逻辑。

## 3. 根因判断

最可能根因：**Nginx 没有把 `/ws/asr` 按 WebSocket 协议转发到后端**。

原因：

1. 当前 Nginx 有 `location /ws/face`，但没有 `location /ws/asr`。
2. `/ws/asr` 请求会落入 `location /`。
3. `location /` 没有：
   - `proxy_http_version 1.1`
   - `proxy_set_header Upgrade $http_upgrade`
   - `proxy_set_header Connection "upgrade"`
   - `proxy_buffering off`
   - 较长的 `proxy_read_timeout/proxy_send_timeout`
4. WebSocket 握手依赖 HTTP/1.1 Upgrade；缺失这些头时，Nginx 会按普通 HTTP 代理转发，后端 Flask-Sock 不能完成 WebSocket 握手，浏览器就会报 `WebSocket connection ... failed`。

这也解释了为什么 `/ws/face` 可以正常，而 `/ws/asr` 失败：`/ws/face` 命中了专门的 Upgrade 配置，`/ws/asr` 没有。

## 4. 为什么不像是 DashScope 公网连接失败

如果是后端服务器无法连接 DashScope 公网 WebSocket，通常链路表现会不同：

```text
浏览器 -> Nginx -> 后端 /ws/asr 握手成功
前端开始发送音频包
后端尝试连接 DashScope 失败
后端返回 {"code": 50000, "msg": "asr_forward_failed", "data": null}
```

当前错误是在浏览器建立 `wss://sx.juanquan.top/ws/asr` 时就失败，控制台没有显示已经连接后收到 `asr_forward_failed`。所以第一失败点更靠前，是浏览器到 Nginx/后端的 WebSocket Upgrade。

不过，在修复 Nginx 后，还需要继续验证服务器到 DashScope 的公网出站能力；这是第二阶段问题，不是当前浏览器握手失败的第一原因。

## 5. 建议的 Nginx 修正

按当前部署要求，保持你提供的现有 Nginx 配置不变，只新增一个 `/ws/asr` 的 WebSocket 代理块。也就是说，`location /` 和现有 `location /ws/face` 都不改，只把 `/ws/asr` 补齐为与 `/ws/face` 同类的 Upgrade 配置。

建议新增的块如下，放在 `server { listen 443 ssl; ... }` 内，位置可放在 `location /ws/face` 后面：

```nginx
    location /ws/asr {
        proxy_pass http://sx;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "Upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_buffering off;
        proxy_read_timeout 300s;
        proxy_send_timeout 300s;
    }
```

## 6. 修复后的验证顺序

### 6.1 验证 Nginx 到后端 WebSocket Upgrade

在任意能访问域名的机器上验证握手是否返回 `101 Switching Protocols`：

```bash
curl -i -N \
  -H "Connection: Upgrade" \
  -H "Upgrade: websocket" \
  -H "Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==" \
  -H "Sec-WebSocket-Version: 13" \
  https://sx.juanquan.top/ws/asr
```

期望看到：

```text
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: upgrade
```

如果还是 200、301、400、404、426、502 或 504，就说明 Nginx/上游转发仍有问题。

### 6.2 验证后端 `/ws/asr` 路由

本地或服务器内网可以用无效 JSON 验证后端 WebSocket 路由是否可达：

```bash
node -e "const ws=new WebSocket('wss://sx.juanquan.top/ws/asr'); ws.onopen=()=>ws.send('not-json'); ws.onmessage=e=>{console.log(String(e.data)); ws.close(); process.exit(0)}; ws.onerror=e=>{console.error('ws error', e); process.exit(1)}"
```

如果握手成功，后端应返回类似：

```json
{"code":40001,"msg":"invalid_json","data":null}
```

这只能证明项目 `/ws/asr` 路由和 Nginx Upgrade 可用，还不能证明 DashScope 上游可用。

### 6.3 验证服务器到 DashScope 公网出站

Nginx 修复后，如果前端能连接但 ASR 仍无转写或收到 `asr_forward_failed`，再检查后端容器/服务器到 DashScope 的出站：

```bash
python - <<'PY'
from websocket import create_connection
ws = create_connection(
    "wss://dashscope.aliyuncs.com/api-ws/v1/inference",
    timeout=10,
    header=["Authorization: Bearer <masked-api-key>"],
)
print("connected")
ws.close()
PY
```

同时看后端日志是否出现：

```text
连接 DashScope ASR 失败
向 DashScope ASR 发送数据失败
DashScope ASR task-started 等待超时
asr_forward_failed
```

如果出现这些日志，才说明进入第二阶段：服务器公网出站、DNS、证书、API Key 或 DashScope 服务连接问题。

## 7. 当前结论

1. 当前服务器 `/ws/asr` 连接失败的首要原因是 Nginx 只给 `/ws/face` 配置了 WebSocket Upgrade，遗漏了 `/ws/asr`。
2. `/ws/asr` 会落入普通 `location /`，该 location 没有 WebSocket 必需的 `proxy_http_version 1.1` 和 Upgrade/Connection 头。
3. 当前浏览器错误发生在连接 `wss://sx.juanquan.top/ws/asr` 阶段，优先处理 Nginx `/ws/` 代理，而不是先改后端代码。
4. 修复 Nginx 后，再验证后端服务器能否连通 DashScope 公网 WebSocket；如果失败，才继续排查服务器出站网络、DNS、证书和 ASR API Key。
