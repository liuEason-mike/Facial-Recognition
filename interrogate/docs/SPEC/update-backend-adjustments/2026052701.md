# `/ws/face` 链路检查结果（2026-05-27）

## 1. 检查范围与限制

本次只检查当前仓库中与 `/ws/face` 直接相关的前端发送、后端 WebSocket、视觉推理、基线/异常评分、数据库写入、部署文档和本机运行环境证据。

未能直接检查生产服务器上的实际 Nginx 配置、正在运行的 Docker Compose、容器启动参数和生产日志。因此，生产 Nginx 是否完全按规范配置、服务器宿主机 GPU 驱动是否安装，只能给出代码和当前本机环境层面的结论；最终仍需要在生产机上执行验证命令。

## 2. 当前主链路

当前 Docker 和 `run.py` 优先托管 `html-interrogate` 前端构建产物，因此 `/ws/face` 主链路应按 `html-interrogate` 判断：

```text
html-interrogate 浏览器摄像头
  -> buildWebSocketUrl('/ws/face')
  -> app/routes/ws/register.py:/ws/face
  -> app/routes/ws/video.py:process_raw_ws_request
  -> app/service/video/realtime_features.py:process_frame_ws_payload
  -> 同一条 WebSocket 回包返回分析结果
```

服务端 `/ws/face` 是单连接串行处理：收到一条消息后调用 `process_video(payload)`，处理完成再 `ws.send(...)`，然后继续接收下一条。证据见 `app/routes/ws/register.py:10-23`。

## 3. 前端实际发帧率

结论：当前主前端目标发帧率是 **5 FPS**，但因为已有 in-flight 背压，实际普通帧发送率是 **不超过 5 FPS**，并会随“后端处理耗时 + 网络/Nginx 往返延迟”自动下降。

证据：

- `html-interrogate/src/constants/session.ts:2` 定义 `FACE_FRAME_RATE = 5`。
- `html-interrogate/src/composables/useFaceAnalysisSocket.ts:84-88` 使用 `setInterval(sendFrame, 1000 / FACE_FRAME_RATE)`，即目标间隔 200ms。
- `html-interrogate/src/composables/useFaceAnalysisSocket.ts:37` 维护 `inFlightFrame`。
- `html-interrogate/src/composables/useFaceAnalysisSocket.ts:55-57` 在已有未回包帧时跳过发送。
- `html-interrogate/src/composables/useFaceAnalysisSocket.ts:74-76` 发送后将 `inFlightFrame = true`。
- `html-interrogate/src/composables/useFaceAnalysisSocket.ts:139-151` 收到回包或解析失败后将 `inFlightFrame = false`。
- `html-interrogate/src/utils/faceFrameGate.ts:6-8` 明确普通帧只有 `!inFlight` 时才允许发送。

因此：

| 场景 | 实际表现 |
| --- | --- |
| 单帧端到端回包小于 200ms | 基本接近 5 FPS |
| 单帧端到端回包约 350ms | 下一次可发送通常落到后续定时器 tick，实际约 2-3 FPS |
| 单帧端到端回包约 800ms | 实际约 1 FPS 左右 |
| 后端或网络长时间不回包 | 前端普通帧暂停发送，不会持续堆积普通视频帧 |

补充：`html-interrogate/src/utils/media.ts:5-17` 会按摄像头真实 `videoWidth/videoHeight` 截图，并以 JPEG `quality = 0.72` 转 Base64；`html-interrogate/src/composables/useMediaDevices.ts:12-18` 摄像头请求 ideal 为 1280x720。因此当前主链路发送的是接近 720p 的 JPEG Base64，未在发送前降采样。网络上 Base64 会比二进制 JPEG 更大。

历史/非主前端中仍存在无背压发帧：

- `html/src/pages/index.vue` 使用 `MAGIC_NUMBER.frameRate = 3`，固定 3 FPS，无 in-flight 背压。
- `html/src/pages/legacy.vue` 默认 `DEFAULT_FRAME_RATE = 15`，可调帧率，无 in-flight 背压。

如果部署或测试时访问的是旧 `html/` 页面而不是 `html-interrogate`，仍可能出现前端持续发帧导致 WebSocket 队列堆积。

## 4. 后端处理耗时来源

`process_frame_ws_payload` 已经在回包中加入分段耗时，字段包括：

```text
processing_ms
performance.decode_ms
performance.face_analyze_ms
performance.face_landmark_ms
performance.emotion_ms
performance.db_ms
performance.baseline_ms
performance.anomaly_ms
performance.total_ms
```

证据见 `app/service/video/realtime_features.py:466-475`、`app/service/video/realtime_features.py:481-545`。

### 4.1 推理链路

单帧主耗时集中在 `face_analyze_ms`：

- `decode_ms`：Base64 解码 + OpenCV 解码，见 `app/service/video/realtime_features.py:481-483`。
- `face_landmark_ms`：MediaPipe FaceLandmarker 检测，见 `app/service/video/face_analyzer.py:167-169`。
- `emotion_ms`：DeepFace/TensorFlow 情绪识别，见 `app/service/video/face_analyzer.py:208-228` 和 `app/service/video/emotion.py:83-112`。
- 情绪识别不是每帧都跑，默认 `FACE_EMOTION_INFER_EVERY_N_FRAMES=5`，其余帧复用缓存，见 `html` 无关的后端配置 `docs/SPEC/env.md:38-39` 和 `app/service/video/face_analyzer.py:117`、`app/service/video/face_analyzer.py:209-227`。

如果服务器部署后延迟明显，但本地 127.0.0.1 基本无延迟，优先看生产回包里的 `performance.face_analyze_ms`、`performance.face_landmark_ms`、`performance.emotion_ms`。这些值高，说明主要是后端 CPU 推理耗时，而不是 Nginx。

### 4.2 数据库写入

数据库写入是同步的，且每个有效分析帧都会走一次 `commit + refresh`：

- `app/service/video/realtime_features.py:502-512` 在 `is_database_enabled()` 为真时写入 `emotion_real_time`。
- `app/repository/emotion_real_time.py:28-35` 每帧 `s.add(obj)`、`s.commit()`、`s.refresh(obj)`。
- `ENABLE_DATABASE` 默认是 `0`，见 `app/config.py:38` 和 `docs/SPEC/env.md:11`。

因此：

- `ENABLE_DATABASE=0` 时，数据库不是 `/ws/face` 延迟来源。
- `ENABLE_DATABASE=1` 且 MySQL 在远端或磁盘慢时，`performance.db_ms` 会直接进入单帧延迟。
- 当前异常表 `emotion_anomaly` 写入已经不在主路径中，`_score_anomaly` 注释说明“当前不写 emotion_anomaly 表”，见 `app/service/video/realtime_features.py:426-430`。

### 4.3 基线训练与异常评分

基线训练已经从同步主路径移到后台线程：

- `FACE_BASELINE_TRAIN_WORKERS` 默认 1 个 worker，见 `app/service/video/realtime_features.py:34-37`。
- `test_status=0` 收集样本后调度后台训练，见 `app/service/video/realtime_features.py:301-336` 和 `app/service/video/realtime_features.py:339-354`。
- `test_status=1` 不再采集样本；当模型 `ready` 时同步执行异常评分，见 `app/service/video/realtime_features.py:523-531`。
- 异常评分使用 5 个 IsolationForest 域模型，见 `app/service/video/isolation_forest/iso_forest_detector.py:279-292`。

因此基线训练本身通常不再阻塞单帧回包，但后台训练仍会占用 CPU，并且 `test_status=1` 且模型 ready 后的 `anomaly_ms` 是同步耗时。

## 5. 延迟来源判断

当前代码层面最可能的部署延迟来源排序：

1. **后端 CPU 推理耗时**：尤其是 MediaPipe FaceLandmarker 和 DeepFace/TensorFlow 情绪识别。看 `face_analyze_ms`、`face_landmark_ms`、`emotion_ms`。
2. **网络传输 + Nginx 往返**：主前端有 in-flight 背压，网络 RTT 不会造成普通帧无限堆积，但会把实际 FPS 从 5 FPS 压低；同时 720p JPEG Base64 上传体积较大。
3. **数据库同步提交**：仅当 `ENABLE_DATABASE=1` 时成立；看 `db_ms`。
4. **异常评分同步耗时**：仅 `test_status=1` 且基线 `ready` 时成立；看 `anomaly_ms`。
5. **WebSocket 队列堆积**：当前 `html-interrogate` 普通帧已有背压，不是主要问题；但旧 `html/` 页面、多个页面同时连接、或其他客户端无背压时仍会堆积。
6. **Nginx 配置错误**：如果 `Upgrade/Connection` 缺失，通常表现为连接失败或断开，而不是单纯变慢。实际生产 Nginx 配置未在仓库中，不能仅凭代码确认。

定位规则：

| 现象 | 判断 |
| --- | --- |
| `processing_ms` 本身很高 | 后端处理慢 |
| `face_analyze_ms` 高 | MediaPipe/姿态/AU/情绪等视觉分析慢 |
| `emotion_ms` 周期性升高 | DeepFace/TensorFlow 情绪推理慢 |
| `db_ms` 高 | 数据库同步写入慢 |
| `anomaly_ms` 高 | 基线 ready 后 IsolationForest 评分慢 |
| `processing_ms` 低但前端显示明显慢 | 网络/Nginx/浏览器侧端到端延迟或旧前端队列 |

## 6. GPU 是否已启用

结论：从当前代码、Dockerfile、部署文档和本机运行环境看，`/ws/face` 后端 **没有启用 GPU**。

证据：

- MediaPipe 初始化只传了 `model_asset_path`，没有设置 GPU delegate，见 `app/service/video/face_analyzer.py:98-107`。
- IsolationForest 属于 scikit-learn CPU 模型，见 `app/service/video/isolation_forest/iso_forest_detector.py:154-177`。
- 情绪识别使用 TensorFlow/DeepFace，但代码没有显式设备选择；只有当运行环境正确安装 CUDA/cuDNN 并且 TensorFlow 能看到 GPU 时才可能自动使用 GPU，见 `app/service/video/emotion.py:10-12`、`app/service/video/emotion.py:95-100`。
- `build/docker/Dockerfile:24-46` 使用普通 Python slim 运行镜像，只安装 OpenCV/MediaPipe 所需系统库，没有 CUDA/cuDNN/NVIDIA runtime 配置。
- `docs/SPEC/deployment.md:37-40` 明确当前是 CPU 可运行部署，后续 GPU/NPU 需要单独补充驱动、运行时和镜像规范。
- `requirement.txt:36` 使用普通 `tensorflow` 依赖，没有单独的 GPU 部署说明。

## 7. 本次结论

1. 当前主前端 `/ws/face` 目标发帧率是 5 FPS，但有 in-flight 背压；实际 FPS 不会超过 5，并会被后端处理耗时和网络 RTT 压低。
2. 当前主前端不存在“无背压持续发普通帧”的问题；旧 `html/` 页面仍存在无背压发帧风险。
3. 服务器部署后延迟明显时，优先查看回包中的 `performance` 字段。若 `processing_ms/face_analyze_ms/emotion_ms` 高，根因在后端 CPU 推理；若这些值低但前端仍慢，再查网络、Nginx 和浏览器侧端到端延迟。
4. 数据库只有在 `ENABLE_DATABASE=1` 时才参与 `/ws/face` 实时帧写入；一旦启用，每帧同步 `commit + refresh`，可通过 `db_ms` 直接确认影响。
5. 当前仓库和本机运行环境没有启用 GPU；即使服务器有 GPU，当前 Dockerfile 和代码也不会自动让 MediaPipe/scikit-learn 使用 GPU。
6. 生产 Nginx 实际配置未在仓库中，不能确认是否与文档一致；需要在服务器核对 `/ws/` 是否配置 `proxy_http_version 1.1`、`Upgrade`、`Connection "upgrade"` 和足够的 read/send timeout。

# 二次分析：服务器 2-3 FPS 是前端 ACK 型背压限速

## 8. 新观测

最新实测现象：

- 本地启用时，链路可达到约 13 FPS。
- 部署到服务器后，前端只能稳定约 2-3 FPS。
- 服务器并不是后端主动丢帧，而是前端发送频率被限制住了。

这说明当前瓶颈至少不只是后端单帧推理能力。更准确地说，`html-interrogate` 当前的前端策略是“上一帧收到后端回包以后，才允许发送下一帧普通帧”。这是一种 stop-and-wait / ACK 型背压。

证据仍是：

- `html-interrogate/src/composables/useFaceAnalysisSocket.ts` 中 `inFlightFrame` 是布尔值。
- 普通帧发送后设置 `inFlightFrame = true`。
- 只有收到 WebSocket 回包或解析失败后才设置回 `false`。
- `shouldSendFaceFrame({ force, inFlight })` 在 `inFlight=true` 时阻止普通帧发送。

这种策略能避免旧帧堆积，但也把发送 FPS 绑定到了“后端处理耗时 + 网络 RTT + Nginx/浏览器 WebSocket 往返时间”。

## 9. 为什么本地高、服务器低

本地 `127.0.0.1` 时，网络 RTT 基本可以忽略。如果后端单帧处理约 70-80ms，后端能力可以接近 12-14 FPS。

部署到服务器后，即使后端单帧处理仍然是 70-80ms，只要端到端回包耗时变成 300-450ms，前端 stop-and-wait 就会自然降到约 2-3 FPS：

```text
当前普通帧实际 FPS ≈ 1000 / 单帧端到端回包毫秒数

回包 80ms  -> 约 12.5 FPS
回包 200ms -> 约 5 FPS
回包 350ms -> 约 2.8 FPS
回包 450ms -> 约 2.2 FPS
```

因此，服务器上看到 2-3 FPS，不一定代表后端只能处理 2-3 FPS；也可能只是前端等待每一帧回包后才发下一帧，网络往返把管道“空出来”的时间放大了。

补充：如果当前部署环境实测能到 13 FPS，需要确认是否使用的是最新 `html-interrogate` 构建产物。当前源码中 `FACE_FRAME_RATE = 5`，理论上主前端定时器目标不应超过 5 FPS；若实测为 13 FPS，可能是使用了旧页面、自定义构建、测试脚本，或本地已改过常量但未体现在当前源码。

## 10. 修改目标

提升 `/ws/face` 处理帧率时，目标不应是简单去掉背压，否则服务器网络稍差时会重新积压旧帧。

推荐目标：

1. 前端按目标 FPS 稳定采集和发送，例如 10-15 FPS。
2. 允许少量在途帧，用小窗口隐藏服务器网络 RTT。
3. 队列必须有上限，超过上限时丢弃旧帧或跳过本次发送。
4. 前端只展示最新 `client_seq` 的结果，过期回包不更新 UI。
5. 服务端可选地只处理最新帧，避免为了追历史帧造成展示延迟。
6. 降低单帧体积，减少 Base64/JPEG 在公网和 Nginx 上的传输压力。

## 11. 推荐改造方案

### 方案 A：前端有界并发窗口（最快止血）

把当前布尔 `inFlightFrame` 改成“在途帧数量窗口”，例如允许 3-5 帧同时在途。

前端改动点：

- `html-interrogate/src/constants/session.ts`
  - 新增 `FACE_TARGET_FRAME_RATE = 12` 或 `15`。
  - 新增 `FACE_MAX_IN_FLIGHT = 4`。
  - 新增 `FACE_MAX_BUFFERED_BYTES`，例如 2-4MB，用于避免浏览器 WebSocket 发送缓冲无限增大。
- `html-interrogate/src/utils/faceFrameGate.ts`
  - 从 `inFlight: boolean` 改成 `inFlightCount: number`。
  - 允许普通帧在 `inFlightCount < FACE_MAX_IN_FLIGHT` 且 `socket.bufferedAmount` 未超阈值时发送。
  - `force=true` 的完成帧仍允许发送，但也要注意连接状态。
- `html-interrogate/src/composables/useFaceAnalysisSocket.ts`
  - 用 `pendingFrames = new Map<client_seq, sent_at>` 或 `inFlightCount` 替代 `inFlightFrame`。
  - 每次发送普通帧后记录 `client_seq` 和发送时间。
  - 收到回包后根据 `client_seq` 删除 pending，并计算 `rtt_ms`。
  - 若回包 `client_seq` 小于最近已展示序号，则丢弃，不更新 UI。
  - 记录 `sent_fps`、`received_fps`、`inFlightCount`、`dropped_by_backpressure`，方便继续排查。

这个方案的收益是：如果后端单帧处理能力约 13 FPS，而服务器 RTT 导致 stop-and-wait 只能 2-3 FPS，那么 3-5 个在途窗口能显著提升后端吃帧速度。简单估算：

```text
后端处理 75ms/帧，端到端回包 350ms：

当前 stop-and-wait:
1 / 350ms ≈ 2.8 FPS

允许 4-5 帧在途后：
后端可以连续读到后续帧，理论上更接近 1 / 75ms ≈ 13 FPS
```

风险：

- Flask-Sock 当前服务端仍是单连接串行 `receive -> process -> send`，在途窗口过大时，旧帧可能排在 WebSocket/socket 缓冲中。
- 如果前端不丢弃旧 `client_seq` 回包，UI 会显示延迟结果。
- 如果不限制 `socket.bufferedAmount`，弱网时浏览器内存和延迟会继续上升。

因此方案 A 必须配套 `client_seq` 过期回包丢弃和 `bufferedAmount` 限制。

### 方案 B：前端有界窗口 + 后端 latest-frame 邮箱（推荐）

这是更适合实时视频分析的方案：前端允许小窗口发送，后端每个连接只保留“最新待处理帧”，旧普通帧可以被覆盖，控制帧优先处理。

后端改动方向：

- 将 `/ws/face` 的单连接处理从纯串行循环升级为“接收线程/循环 + 处理循环”。
- 接收侧持续 `ws.receive()`，解析后：
  - `test_status=0/1` 普通分析帧进入 `latest_payload`，覆盖旧普通帧。
  - `test_status=2` 结束帧或控制帧进入优先队列，不能被普通帧覆盖。
- 处理侧每次取最新普通帧调用 `process_video(payload)`，处理完成后回包。
- 每个连接维护最大队列长度 1 或 2，避免历史帧堆积。
- 断开连接时释放该连接的线程、队列和状态。

前端仍保留：

- `FACE_MAX_IN_FLIGHT` 小窗口。
- `client_seq` 过期回包丢弃。
- `socket.bufferedAmount` 上限。
- 发送/接收 FPS 和 RTT 指标。

这个方案能同时满足：

- 网络 RTT 不再把前端发送压到 2-3 FPS。
- 后端不会为了处理旧帧造成画面和分析结果越来越滞后。
- UI 始终更接近实时状态。

这是当前最推荐的改法。

### 方案 C：降低单帧传输体积（应和 A/B 同时做）

当前 `captureVideoFrame` 使用摄像头真实分辨率截图，摄像头请求 ideal 是 1280x720，并以 JPEG `quality=0.72` 转 Base64。公网环境下这会增加上传耗时和 Nginx/WebSocket 传输压力。

建议改动：

- `html-interrogate/src/utils/media.ts`
  - 将 `captureVideoFrame(video, quality)` 改成支持 `maxWidth/maxHeight`。
  - 默认将发送帧缩到 640x360 或 720p 以下，例如宽度 640。
  - JPEG quality 从 `0.72` 调到 `0.55-0.65` 之间测试。
- `html-interrogate/src/composables/useMediaDevices.ts`
  - 可以保留本地预览 1280x720，但发送前降采样。
- 后续可评估二进制 WebSocket 传 JPEG Blob/ArrayBuffer，替代 JSON Base64。

Base64 相比二进制会额外膨胀约三分之一。仅将 1280x720 改成 640x360，就可能把单帧像素数降到 1/4，再叠加 JPEG quality 下调，公网发送压力会明显下降。

### 方案 D：长期协议优化

如果后续要稳定 15 FPS 以上，建议考虑更大的协议升级：

- WebSocket 二进制帧传输 JPEG/WEBP，JSON 只传元数据或用小头部协议。
- 前端使用 `OffscreenCanvas` 或 Worker 编码，避免主线程 `canvas.toDataURL()` 阻塞。
- 后端按二进制消息直接解码，不再走 Base64 JSON。
- 如果要接近视频流，应评估 WebRTC 或专门的视频流协议，再由后端抽帧分析。

这类改动收益更大，但协议、测试和联调成本也更高，不建议作为第一步止血。

## 12. 建议实施顺序

第一步先做方案 A + C：

1. 前端把 `inFlightFrame` 从布尔值改成有界窗口。
2. 目标发送帧率先设为 12 FPS，`FACE_MAX_IN_FLIGHT` 先设为 4。
3. 增加 `socket.bufferedAmount` 限制。
4. 前端按 `client_seq` 丢弃过期回包。
5. 发送帧降采样到 640 宽、JPEG quality 先设为 0.6。
6. UI 或运行日志增加发送 FPS、接收 FPS、RTT、pending 数、丢弃数。

第二步再按实测决定是否做方案 B：

- 如果服务器实际接收/处理能上去，但 UI 偶尔显示旧结果，继续收紧前端过期回包丢弃。
- 如果服务端 WebSocket 队列仍积压，做后端 latest-frame 邮箱。
- 如果 `performance.face_analyze_ms` 仍然高，再优化模型推理、GPU、数据库或进程并发。

## 13. 验收指标

改造后建议按以下指标验收：

| 指标 | 目标 |
| --- | --- |
| 前端发送 FPS | 服务器环境稳定 8-12 FPS 起步 |
| 前端接收 FPS | 接近后端实际处理能力，允许低于发送 FPS |
| 平均 RTT | 可观测并低于改造前 |
| pending 帧数量 | 正常在 `FACE_MAX_IN_FLIGHT` 内波动 |
| `socket.bufferedAmount` | 不持续增长 |
| UI 最新序号 | 只展示最新 `client_seq`，不被旧回包覆盖 |
| 后端 `processing_ms` | 与改造前基本一致或下降 |
| 端到端主观延迟 | 不能随运行时间持续累积 |

## 15. 当前推荐结论

当前问题的第一优先级不是继续检查后端是否丢帧，而是把前端 stop-and-wait 背压改成“有界并发 + 过期回包丢弃 + 降采样传输”。

推荐第一版参数：

```text
FACE_TARGET_FRAME_RATE = 12
FACE_MAX_IN_FLIGHT = 4
FACE_MAX_BUFFERED_BYTES = 2 * 1024 * 1024
FACE_CAPTURE_MAX_WIDTH = 640
FACE_CAPTURE_JPEG_QUALITY = 0.6
```

这组参数能先验证服务器 2-3 FPS 是否主要由前端等待回包造成。若改造后服务器环境接收 FPS 明显上升，说明判断成立；若仍上不去，再看 `performance` 分段耗时和 Nginx/上传带宽。
