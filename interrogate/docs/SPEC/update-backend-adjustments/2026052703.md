### 结论

浏览器中看到的 `POST http://localhost:5000/api/asr/diarization/align` 是符合当前架构的请求地址。这个地址是项目后端提供给前端调用的说话人分离对齐接口，不应该由前端直接请求 `192.168.1.12:6188` 或 `8.140.17.203:6188`。

当前链路应为：

```text
前端 html-interrogate
  -> 项目后端 http://localhost:5000/api/asr/diarization/align
  -> 外部 pyannote 服务 PYANNOTE_DIARIZATION_ENDPOINT
     例如 http://192.168.1.12:6188/diarize
     或   http://8.140.17.203:6188/diarize
```

因此，`503 SERVICE UNAVAILABLE` 不表示前端请求错了 IP，而表示项目后端已经收到 `/api/asr/diarization/align` 请求，但后端调用外部 pyannote 说话人分离服务失败，所以返回了 `speaker_diarization_unavailable`。

### 代码核对

1. 前端调用位置：

   `html-interrogate/src/api/asrDiarization.ts`

   前端通过 `getApiBaseUrl()` 生成项目后端地址，然后请求：

   ```text
   /asr/diarization/align
   ```

   本地开发时 `VITE_API_BASE_URL` 默认等价于 `/api`，所以完整请求会落到：

   ```text
   http://localhost:5000/api/asr/diarization/align
   ```

2. 后端路由位置：

   `app/routes/api/asr_diarization.py`

   后端注册的路由是：

   ```text
   POST /api/asr/diarization/align
   ```

   该接口负责接收前端上传的音频窗口和 ASR segments，然后调用 `speaker_diarization_service.diarize_and_align(...)`。

3. 外部 pyannote 服务调用位置：

   `app/service/audio/speaker_diarization.py`

   `PyannoteDiarizationClient` 读取环境变量：

   ```text
   PYANNOTE_DIARIZATION_ENDPOINT
   ```

   如果该变量未配置，会抛出：

   ```text
   endpoint_required
   ```

   后端捕获 `PyannoteDiarizationError` 后统一返回：

   ```text
   HTTP 503
   speaker_diarization_unavailable
   ```

### 最可能原因

本次 `503` 的最可能原因是后端运行环境中 `PYANNOTE_DIARIZATION_ENDPOINT` 未配置、配置不完整，或项目后端无法访问该地址。

需要特别注意：配置值必须是外部 pyannote 的完整 HTTP 接口地址，通常要包含 `/diarize`，例如：

```bash
PYANNOTE_DIARIZATION_ENDPOINT=http://8.140.17.203:6188/diarize
```


如果只配置成 `http://192.168.1.12:6188` 或 `http://8.140.17.203:6188`，而外部服务实际只在 `/diarize` 提供接口，则后端可能收到外部服务的 `404`，随后项目后端仍会对前端返回 `503`。


### 新增根因判断

当前问题不再是 `PYANNOTE_DIARIZATION_ENDPOINT` 的 URL 写错，也不是 `192.168.1.12:8188/diarize` 服务不可达。

本次检查发现运行环境中存在代理变量：

```text
HTTP_PROXY=http://127.0.0.1:7890
HTTPS_PROXY=http://127.0.0.1:7890
http_proxy=http://127.0.0.1:7890
https_proxy=http://127.0.0.1:7890
NO_PROXY=127.0.0.1,192.168.*,10.*,127.*,localhost,<local>
```

`curl --noproxy '*'` 可以绕过代理直连成功，但 Python 的 `urllib.request.urlopen` 会读取系统代理变量。当前 `NO_PROXY=192.168.*` 这种通配写法在 `urllib` 场景下没有稳定绕过内网地址，导致后端请求可能走到 `127.0.0.1:7890` 代理，再表现为 `TimeoutError`，最终被接口包装成：

```text
speaker_diarization_unavailable
```

### 处理建议

1. 后端进程启动环境中，给 pyannote 所在主机配置精确的代理绕过：

   ```bash
   NO_PROXY=127.0.0.1,localhost,192.168.1.12
   no_proxy=127.0.0.1,localhost,192.168.1.12
   ```

   如果还需要保留其他内网段，建议同时保留精确 IP：

   ```bash
   NO_PROXY=127.0.0.1,localhost,192.168.1.12,192.168.1.0/24
   no_proxy=127.0.0.1,localhost,192.168.1.12,192.168.1.0/24
   ```

   对 Python/urllib 来说，最稳的是写入具体主机 IP：`192.168.1.12`。

2. 如果后端进程不需要代理访问外网，可以在启动后端前直接取消代理变量：

   ```bash
   unset HTTP_PROXY HTTPS_PROXY http_proxy https_proxy ALL_PROXY all_proxy
   ```

   然后重启后端。

3. 如果使用 systemd、supervisor、pm2、docker 或 shell 脚本启动后端，需要把上述 `NO_PROXY/no_proxy` 写入实际启动配置，而不是只写在当前终端里。

4. 服务器上再次复测时，应优先看 Flask 日志：

   ```text
   说话人分离服务暂不可用: TimeoutError
   ```

   如果是 `TimeoutError`，优先按代理绕过处理；如果仍是 `endpoint_required`，说明实际运行的后端进程没有读到 `.env`，需要确认是否使用了 `interrogation_env` 启动、`python-dotenv` 是否安装、以及修改 `.env` 后是否重启了后端。
