# 说话人分离 pyannote endpoint 代码兜底修改方案

## 1. 问题背景

服务器页面请求：

```text
POST https://192.168.1.10:7116/api/asr/diarization/align
Status Code: 503 SERVICE UNAVAILABLE
```

该请求地址不是错误接口。当前前端设计为只调用项目后端：

```text
前端 html-interrogate
  -> 项目后端 /api/asr/diarization/align
  -> 外部 pyannote 服务 /diarize
```

因此浏览器中看到 `/api/asr/diarization/align` 是符合架构的。`503 speaker_diarization_unavailable` 表示项目后端收到音频窗口后，调用外部 pyannote 服务失败。

当前 Docker 镜像中没有 `.env` 是预期行为：

- `.dockerignore` 排除了 `.env` 和 `.env.*`。
- Dockerfile 只复制后端代码、模型文件、`run.py` 和前端构建产物。
- 生产推荐通过 `docker-compose.yml` 的 `env_file` 或 `environment` 注入运行时环境变量。

如果服务器容器没有注入：

```dotenv
PYANNOTE_DIARIZATION_ENDPOINT=http://192.168.1.12:8188/diarize
```

后端会在调用 pyannote 前判断 endpoint 为空，并返回 `503 speaker_diarization_unavailable`。

## 2. 修改目标

本次修改目标是在服务器运行环境没有正确注入 `PYANNOTE_DIARIZATION_ENDPOINT` 时，后端仍能使用固定的内网 pyannote 服务地址作为兜底：

```text
http://192.168.1.12:8188/diarize
```

修改后预期：

- 前端仍请求 `https://192.168.1.10:7116/api/asr/diarization/align`。
- 后端优先读取 `PYANNOTE_DIARIZATION_ENDPOINT`。
- 如果环境变量和项目 `.env` 都不存在，后端使用代码内置默认地址。
- 外部 pyannote 可用时，`/api/asr/diarization/align` 不再因为 `endpoint_required` 返回 503。
- pyannote 服务真正不可达、超时或返回错误时，仍按当前降级逻辑返回 `speaker_diarization_unavailable`，不影响实时 ASR 主链路。

## 3. 方案对比

### 方案 A：只修服务器 env 注入

在宿主机部署目录 `.env` 或 `docker-compose.yml` 中加入：

```dotenv
PYANNOTE_DIARIZATION_ENDPOINT=http://192.168.1.12:8188/diarize
```

优点：

- 符合“外部服务地址来自环境变量”的配置规范。
- 不需要重新构建镜像，只需要重建容器。
- 后续更换 pyannote 地址不需要改代码。

缺点：

- 如果服务器发布流程经常漏掉 `.env` 或没有正确挂载 env，问题会反复出现。

### 方案 B：前端直接请求 pyannote

将前端请求从 `/api/asr/diarization/align` 改为 `http://192.168.1.12:8188/diarize`。

不建议采用：

- 会绕过后端 ASR segment 对齐逻辑，前端拿不到当前契约中的 `speaker`、`start`、`end`、`text` 结果。
- 浏览器 HTTPS 页面直接请求 HTTP 内网服务可能触发混合内容、CORS 或网络访问问题。
- 会把外部服务拓扑暴露给浏览器，不符合当前前后端契约。

### 方案 C：后端环境变量优先，代码内置默认 endpoint 兜底

在 `app/service/audio/speaker_diarization.py` 中新增默认地址常量，并让 `PyannoteDiarizationClient` 的 endpoint 解析顺序变为：

```text
显式构造参数 endpoint
  -> 进程环境变量 PYANNOTE_DIARIZATION_ENDPOINT
  -> 项目 .env / ENV_FILE / .env.<APP_PROFILE>
  -> DEFAULT_PYANNOTE_DIARIZATION_ENDPOINT
```

推荐采用该方案作为短期稳定修复。

优点：

- 服务器漏注入 env 时仍可访问当前固定 pyannote 服务。
- 保留环境变量覆盖能力，后续迁移 pyannote 地址不需要强制改代码。
- 改动集中在 pyannote 客户端适配层，不影响前端和 ASR WebSocket 主链路。

缺点：

- 代码中出现内网服务地址，后续环境变化需要同步维护。
- 从配置规范角度看，这是生产兜底，不应扩展到密钥、数据库地址或其他敏感配置。

## 4. 推荐修改范围

### 4.1 后端服务适配层

修改文件：

```text
app/service/audio/speaker_diarization.py
```

建议新增常量：

```python
DEFAULT_PYANNOTE_DIARIZATION_ENDPOINT = "http://192.168.1.12:8188/diarize"
```

建议调整初始化：

```python
self.endpoint = endpoint if endpoint is not None else _resolve_env_value(
    "PYANNOTE_DIARIZATION_ENDPOINT",
    DEFAULT_PYANNOTE_DIARIZATION_ENDPOINT,
)
```

建议调整 `_current_endpoint()`：

```python
resolved = _resolve_env_value(
    "PYANNOTE_DIARIZATION_ENDPOINT",
    self.endpoint or DEFAULT_PYANNOTE_DIARIZATION_ENDPOINT,
)
```

保留已有行为：

- 显式传入 `endpoint` 的测试或调用仍优先。
- `PYANNOTE_API_KEY` 仍只从环境变量或 `.env` 读取，不做代码默认值。
- `PYANNOTE_TIMEOUT_SECONDS` 仍默认 `15`。
- pyannote 请求仍使用无代理直连 opener。

### 4.2 文档

需要同步更新：

```text
docs/SPEC/env.md
docs/api/asr-diarization.md
docs/changelog.md
```

更新重点：

- `PYANNOTE_DIARIZATION_ENDPOINT` 从“必须由运行环境提供”调整为“推荐由运行环境提供；缺失时当前版本代码默认使用 `http://192.168.1.12:8188/diarize`”。
- 明确默认地址是当前部署兜底，不是长期替代生产配置。
- 说明前端仍请求 `/api/asr/diarization/align`，不直接请求 pyannote。

### 4.3 测试

修改文件：

```text
test/test_speaker_diarization.py
```

建议增加测试：

1. 进程环境变量缺失、项目 `.env` 缺失时，客户端使用默认 endpoint。
2. 进程环境变量存在时，环境变量覆盖默认 endpoint。
3. 显式构造参数 `endpoint` 存在时，显式参数优先于默认 endpoint。

保留现有测试：

- multipart 请求体构造。
- 无代理直连 opener。
- 请求前重新解析 env。
- API 路由成功和错误降级。

## 6. 风险与回滚

风险：

- 默认 endpoint 写在代码中，当前只适配 `192.168.1.12:8188/diarize` 这一套部署环境。
- 如果服务器无法访问 `192.168.1.12`，仍会返回 503，但错误会从 `endpoint_required` 变为 `URLError`、`TimeoutError` 或连接失败。
- 如果 pyannote 服务迁移，必须通过环境变量覆盖默认值，或同步修改代码默认值后重新构建镜像。

回滚：

- 删除 `DEFAULT_PYANNOTE_DIARIZATION_ENDPOINT` 兜底逻辑。
- 恢复 `PYANNOTE_DIARIZATION_ENDPOINT` 必须由环境变量或 `.env` 提供。
- 重建镜像并重启容器。

## 7. 最终建议

本次可以按方案 C 实施：在代码中加入 `http://192.168.1.12:8188/diarize` 作为 pyannote endpoint 默认兜底，同时保留环境变量优先级。

不要把前端请求改成 pyannote 地址。前端继续请求：

```text
/api/asr/diarization/align
```

后端负责调用：

```text
http://192.168.1.12:8188/diarize
```

这样既能绕开服务器 Docker env 注入遗漏导致的 `endpoint_required`，又不破坏当前前后端契约和 ASR 文本对齐逻辑。
