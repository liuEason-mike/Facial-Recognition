# ASR 说话人分离功能规范

日期：2026-05-25

状态：已实现

范围：`interrogate` 后端 ASR 链路、外部 pyannote 说话人分离服务适配、`html-interrogate` 前端转写展示。

## 1. 背景

当前审讯页面已经具备以下能力：

- 实时音频流采集和音频波形展示。
- `qwen-asr` / DashScope ASR 实时转写。
- “最新转写”展示。
- ASR 关键词提炼。

本次需求是在不破坏上述主链路的前提下，新增多人声区分（Speaker Diarization）能力。说话人分离使用 `pyannote/speaker-diarization-3.1`，但模型不在当前项目本地运行；当前项目通过 HTTP/API 调用另一台服务器上的 pyannote 服务。

## 2. 目标

1. 通过外部 pyannote 服务识别同一段审讯音频中的说话人时间段。
2. 后端将 pyannote 返回的说话人时间段与 qwen-asr 转写片段对齐，输出带 `speaker`、`start`、`end`、`text` 的说话人转写结果。
3. 前端在“最新转写”模块下方新增“说话人转写”区域，按时间顺序展示每段说话内容。
4. 说话人标识使用 `A`、`B`、`C` 等稳定标签，并在前端使用固定颜色区分。
5. pyannote 服务失败、超时或返回空结果时，不影响实时 ASR、最新转写、音频波形和关键词提炼。

## 3. 非目标

本次不做以下修改：

- 不改变实时音频流展示逻辑。
- 不改变“最新转写”模块现有展示逻辑。
- 不改变“关键词提炼”模块现有逻辑。
- 不在当前项目本地部署或加载 `pyannote/speaker-diarization-3.1` 模型。
- 不承诺识别真实身份姓名，只输出会话内稳定的说话人标签，例如 `A`、`B`、`C`。
- 不要求 pyannote 负责转写文本；文本仍由 qwen-asr / DashScope ASR 提供。

## 4. 总体方案

推荐链路：

```text
浏览器麦克风
  -> 现有 ASR 采集和 qwen-asr 实时转写
  -> 后端缓存或接收一段音频窗口
  -> 调用外部 pyannote HTTP/API
  -> 获得 speaker 时间段
  -> 与 ASR segments 按时间重叠对齐
  -> 返回 speaker transcript segments
  -> 前端“说话人转写”区域展示
```

关键约束：

- qwen-asr 继续负责实时转写和 `text` 生成。
- pyannote 仅负责多人声分离和时间段识别。
- speaker diarization 可以允许轻微延迟，不阻塞实时转写。
- 说话人结果是 ASR 的旁路增强结果，不是 ASR 主链路的前置依赖。

## 5. 后端契约

### 5.1 外部 pyannote 服务调用

外部服务配置应通过环境变量注入，不写死在代码中：

| 变量 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `PYANNOTE_DIARIZATION_ENDPOINT` | 是 | 无 | pyannote HTTP/API 地址 |
| `PYANNOTE_API_KEY` | 否 | 空 | pyannote 服务鉴权 token，如服务不需要鉴权可为空 |
| `PYANNOTE_TIMEOUT_SECONDS` | 否 | `15` | 单次 diarization 请求超时时间 |
| `PYANNOTE_SPEAKER_COUNT_HINT` | 否 | `2` | 默认说话人数提示，仅作为 hint，不强制 |
| `PYANNOTE_AUDIO_WINDOW_SECONDS` | 否 | `15` | 每次提交给 pyannote 的音频窗口长度 |

后端适配层按当前已确认的外部服务契约使用 `multipart/form-data`：

```bash
curl -X POST "http://<pyannote-host>:6188/diarize" -F "file=@test.wav"
```

外部服务返回结构：

```json
{
  "code": 0,
  "message": "success",
  "data": [
    {
      "speaker": "SPEAKER_00",
      "start": 0.942,
      "end": 3.305,
      "duration": 2.362
    }
  ]
}
```

当前项目对前端暴露统一接口 `POST /api/asr/diarization/align`，前端上传 WAV 音频窗口和 ASR segments，后端负责调用外部 pyannote、归一化 speaker 标签并做文本对齐。

### 5.2 后端归一化结果

后端返回给前端的说话人转写结果必须至少包含：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `speaker` | string | 是 | 说话人 ID，例如 `A`、`B`、`C` |
| `start` | string | 是 | 展示用开始时间，格式 `HH:mm:ss` |
| `end` | string | 是 | 展示用结束时间，格式 `HH:mm:ss` |
| `text` | string | 是 | 与该时间段对齐后的 ASR 转写内容 |

建议同时保留以下扩展字段，便于排序、调试和后续持久化：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `start_sec` | number | 否 | 相对本轮音频的开始时间，单位秒 |
| `end_sec` | number | 否 | 相对本轮音频的结束时间，单位秒 |
| `confidence` | number | 否 | pyannote 说话人分离置信度 |
| `source` | string | 否 | 固定为 `pyannote` |
| `session_id` | string | 否 | 审讯会话 ID |
| `suspect_id` | string | 否 | 嫌疑人 ID |

示例：

```json
[
  {
    "speaker": "A",
    "start": "11:38:44",
    "end": "11:38:45",
    "start_sec": 0.5,
    "end_sec": 1.8,
    "confidence": 0.86,
    "text": "今天晚上过去。"
  },
  {
    "speaker": "B",
    "start": "11:38:46",
    "end": "11:38:48",
    "start_sec": 2.0,
    "end_sec": 4.1,
    "confidence": 0.82,
    "text": "我知道了。"
  }
]
```

### 5.3 ASR 文本对齐规则

后端需要把 pyannote 时间段和 ASR 片段对齐：

- 若 ASR 片段有 `start`、`end` 秒级字段，按时间重叠面积最大者匹配。
- 若多个 ASR 片段落入同一 speaker 时间段，按时间顺序拼接文本。
- 若一个 ASR 片段跨多个 speaker 时间段，优先按最大重叠时间归属；后续可扩展为拆分片段。
- 若找不到可匹配文本，返回空 `text` 或跳过该段，由前端展示空状态，不得影响实时 ASR。
- 若 pyannote 返回 speaker 标签不是 `A/B/C`，后端统一映射为会话内稳定标签。

## 6. 前端展示要求

### 6.1 区域位置

在“最新转写”模块下方新增“说话人转写”区域。

保持现有模块顺序：

```text
最新转写
说话人转写
关键词提炼
```

如果现有页面结构中“关键词提炼”已在“最新转写”模块内，应将“说话人转写”插入两者之间，不改变关键词提炼的触发逻辑和展示逻辑。

### 6.2 展示格式

每条说话人转写按时间顺序展示：

```text
[A] 11:38:44 - 11:38:45
今天晚上过去。

[B] 11:38:46 - 11:38:48
我知道了。
```

展示要求：

- `[A]`、`[B]`、`[C]` 显示在时间戳附近。
- 同一个 speaker 在当前会话内颜色固定。
- 不同 speaker 使用不同颜色。
- 默认最多展示最近若干条，避免面板高度被撑爆；具体数量实现时按现有右侧面板高度裁剪。
- 没有结果时显示空状态，例如“等待说话人分离结果”。
- pyannote 失败时显示非阻塞提示，例如“说话人分离暂不可用”，但不清空实时转写。
- 按照倒顺序，只展示5条最新的音频


建议颜色映射：

| Speaker | 颜色语义 |
| --- | --- |
| `A` | 蓝色 |
| `B` | 绿色 |
| `C` | 橙色 |
| `unknown` | 灰色 |

## 7. 错误处理与降级

pyannote 相关错误必须降级处理：

- 外部服务超时：记录 warning 日志，前端 speaker 区域显示暂不可用。
- 外部服务 4xx / 5xx：后端不向实时 ASR 主链路抛出致命错误。
- 返回空 `segments`：前端保留空状态，不影响最新转写。
- 返回字段缺失：后端适配层过滤非法 segment，并记录字段名。
- 音频窗口不足：不调用 pyannote 或返回空结果，等待下一窗口。

日志要求：

- 记录 pyannote 请求开始、结束、耗时、segment 数量和错误类型。
- 不记录完整音频 Base64、原始二进制音频、API Key、Token 或隐私原文。

## 9. 验收标准

### 9.1 功能验收

- 实时音频波形展示保持可用。
- “最新转写”仍实时展示 qwen-asr 文本。
- “关键词提炼”仍按现有逻辑触发和展示。
- 有 pyannote 结果时，“说话人转写”按时间顺序展示每段内容。
- speaker 标签显示为 `A/B/C`，且同一 speaker 颜色稳定。
- pyannote 返回失败或超时时，实时转写和关键词提炼不受影响。

### 9.2 接口验收

- 后端返回 speaker segment 时至少包含 `speaker`、`start`、`end`、`text`。
- `start`、`end` 展示格式为 `HH:mm:ss`。
- 同一接口或消息类型不得混用不兼容字段结构。
- 错误响应不暴露堆栈、服务内网地址、API Key、Token、音频 Base64 或原始音频内容。

### 9.3 测试建议

后端：

- pyannote 适配层单元测试：正常返回、空结果、字段缺失、超时、4xx/5xx。
- ASR 对齐单元测试：单段匹配、多段拼接、跨 speaker 片段、无匹配文本。
- WebSocket/API 测试：speaker segment 响应结构和错误降级。

前端：

- 类型和工具测试：speaker 颜色映射、时间排序、空状态。
- 组件测试或结构测试：`RecordingStatus` 中存在“说话人转写”区域。
- 构建验收：`pnpm test:unit`、`pnpm typecheck`、`pnpm build`。

## 10. 已确认实现取舍

1. 外部 pyannote 服务按 HTTP multipart `file=@*.wav` 调用，真实地址通过 `PYANNOTE_DIARIZATION_ENDPOINT` 注入。
2. 前端按固定音频窗口实时上传 WAV，当前窗口长度为 15 秒；审讯结束后批量重跑不在本次范围内。
3. 前端只展示 `speaker`、`start`、`end`、`text`，`confidence` 保留为扩展字段但不直接展示。
4. speaker 标签保持 `A/B/C`，本次不人工映射为“讯问员 / 嫌疑人 / 其他人员”。
